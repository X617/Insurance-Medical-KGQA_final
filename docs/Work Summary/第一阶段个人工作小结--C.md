### 一、 本阶段工作内容

本阶段作为项目的起步期，我的核心任务是完成 GraphRAG 问答主链路的骨架搭建与架构设计。在明确业务需求和系统设计的基础上，我主要完成了以下三项工作：

**（一） 构建高可扩展的 GraphRAG 核心流水线架构** 

在 `src/graph_rag/` 目录下，我从零设计并划分了三大核心子模块：`query_understanding.py`（意图识别与查询重写）、`graph_retriever.py`（图谱检索）以及 `llm_integration.py`（大模型上下文融合）。在设计之初，我重点贯彻了**模块化解耦**的工程思想，通过定义抽象的接口层，确保了未来底层的图数据库（Neo4j）或大模型基座（如通义千问、DeepSeek）进行切换时，核心的 RAG 业务流水线不会受到任何侵入性影响。

**（二） 核心业务链路 Mock 与 API 层的协同打通** 

配合后端的 API 骨架开发，我在 `rag_engine.py` 中实现了 `chat(query, history)` 主控流程的桩代码（Mock）。为了支撑前端“知识溯源卡片”这一核心创新点的展示，我主导设计了 RAG 模块的标准输出数据结构（强制包含 `answer`, `context`, `intent`, `rewritten_query` 等字段）。这一数据契约不仅打通了从底层检索到前端渲染的完整调用链，也为后续替换真实检索逻辑提供了标准化容器。

**（三） 图检索策略预研与 Cypher 样板库构建** 

针对“保险+医养”交叉领域的复杂问答场景，我协同图谱构建负责人，提前梳理了 GraphRAG 检索所需的关键节点字段与关系类型。同时，我按用户意图分类（如“疾病-报销范围-保险产品”、“疾病-护理需求-养老机构”），编写了一套基础的 Cypher 多跳查询样板库，为下一阶段将自然语言转化为图数据库的高效查询奠定了坚实基础。

### 二、 遇到的问题及解决思路

**1. 并发开发中的“数据契约”断层与结构对齐问题**

* **问题描述：** 在初期与后端API及图谱构建模块进行联调时，由于底层图谱 Schema 尚未完全固化，导致 RAG 模块输出的 Mock 数据结构经常与图谱节点的真实结构产生偏差，进而在前端渲染时引发组件级崩溃。

* **解决思路：** 我在团队内部推动并制定了严格的“数据传输对象 (DTO) 契约”。拉会梳理并锁定第一版核心实体类型和边关系后，我在 `graph_retriever.py` 中增加了针对 Mock 数据的 Schema 校验逻辑。这不仅彻底解耦了前后端的开发进度，也提前验证了系统设计说明书中要求的“UI容错渲染机制”。

**2. 医疗专有名词检索发散与大模型“幻觉”隐患**

- **问题描述：** 在设计 `query_understanding` 的底层逻辑时我发现，医疗与保险领域的专有名词壁垒极高，而用户的实际 Query 往往非常口语化。如果直接将口语化文本投入图谱匹配，极易导致检索上下文为空，从而迫使大模型在问答时产生严重的“幻觉（Hallucination）”。
- **解决思路：** 引入了 **Query Rewriting (查询重写) 与实体对齐**理念。我在系统架构前端预留了语义规范化处理层，确保用户的口语化输入在进入检索器之前，先被抽取并映射为标准的医学/保险术语（如将“血压高”对齐为“原发性高血压”）。从架构源头约束了检索的精准度，大幅降低了模型的幻觉率。

### 三、 下一阶段计划

在第一阶段成功跑通项目骨架的基础上，下一阶段我将重点推进 RAG 链路由 Mock 向真实业务逻辑的转化：

1.**实体抽取与意图识别逻辑落地：** 将 `query_understanding.py` 中的占位逻辑替换为真实的解析器，探索采用轻量级正则规则配合 NER 方案，实现对用户输入中“疾病、药品、机构、险种”等核心实体的精准捕获与意图分类。

2.**图谱真实检索与动态上下文拼装：** 引入 Neo4j Python Driver，将编写好的 Cypher 样板与真实数据库对接。实现将图谱检索召回的多跳三元组数据，转化为结构化、高信息密度的纯文本上下文（Context）。

3.**大模型融合与 Prompt 鲁棒性调优：** 真实接入云端大模型（阿里云百炼平台）API，进行深度的 System Prompt 提示词工程调优。重点确保模型在长文本和复杂语境下，能够严格遵循给定的图谱知识生成回答，并精准输出前端所需的“知识溯源依据”。
